# 使用方法（給完全沒碰過 Business Central 的人）

> 假設你完全沒接觸過 Microsoft Dynamics 365 Business Central / Navision。
> 這份文件會手把手帶你跑起整個專案，並對應到原始碼位置。
> 全部指令以 **Windows 11 + PowerShell** 為例。

---

## 0. 先把名詞講清楚

| 名詞 | 中文白話 |
|------|---------|
| **Microsoft Dynamics 365 Business Central**（簡稱 **BC**） | Microsoft 賣給中小企業的雲端 ERP（管財務、採購、庫存、銷售）。前身叫 **Navision** / **NAV**。 |
| **AL** | BC 專用的程式語言。要客製 BC 一定用它。語法像 Pascal。 |
| **Extension** | BC 的「外掛」。我們不直接改 Microsoft 原始碼，而是寫一個 Extension 加上去。 |
| **Codeunit** | AL 裡的「商業邏輯類別」（沒有畫面）。 |
| **Page** | AL 裡的「畫面」。 |
| **API Page** | 一種特別的 Page，會自動變成一個 RESTful API endpoint，給外部系統呼叫。 |
| **OData** | 一種標準化的 REST API 規範。BC 對外的 API 全都長 OData 樣子。 |
| **MSP** | Managed Service Provider，幫客戶導入 / 維運系統的服務商。 |
| **Sandbox** | BC 提供的測試環境，免費可申請。 |

## 1. 這個專案到底在做什麼？

模擬一個情境：

> 我是 MSP 工程師，客戶想要在 Business Central 內加一個「採購請購單核准流程」，並且要把核准後的請購單自動轉成正式採購單（Purchase Order），同時要有一個給內部員工看的 **Web Dashboard**。

所以我做了：

1. 一個 **AL Extension**（住在 BC 裡）— [`extensions/bc-procurement/`](extensions/bc-procurement/)
2. 一個 **Python FastAPI 後端**（外部系統的整合中樞）— [`apps/api/`](apps/api/)
3. 一個 **React Dashboard**（內部員工用）— [`apps/web/`](apps/web/)
4. **CI/CD**、**Docker**、**Audit Log**、**Swagger 文件**……整套都包好。

## 3. 從 PowerShell 啟動專案

打開 PowerShell，移動到專案根目錄：

```powershell
cd C:\Users\Xu\D365\Business-Central-Integration-Suite
```

### 3.1 一鍵初始化（推薦第一次跑）

```powershell
.\scripts\init-dev.ps1
```

這個 script 會：
- 在 `apps/api/.venv/` 建立 Python 虛擬環境
- `pip install -r requirements.txt`
- 在 `apps/web/` 跑 `npm install`
- 從 `.env.example` 複製出 `.env`
- 建立 SQLite 資料庫並塞 4 筆 seed data

> 第一次跑 npm install 大約 1–3 分鐘；等它跑完。

### 3.2 啟動 Backend（FastAPI）

開一個 PowerShell 視窗：

```powershell
cd C:\Users\Xu\D365\Business-Central-Integration-Suite\apps\api
.\.venv\Scripts\Activate.ps1
uvicorn app.main:app --reload
```

成功的話會看到：

```

INFO:     Started reloader process ...
INFO:     Application startup complete.
```

打開瀏覽器去 `http://localhost:8000/health`，應該會看到：

```json
{"status":"ok","bc_mode":"mock","version":"1.0.0"}
```

### 3.3 啟動 Frontend（React + Vite）

**另外開一個** PowerShell 視窗（不要關 backend）：

```powershell

```

成功的話會看到：

```
VITE v5.4.x  ready in ...
➜  Local:   http://localhost:5173/
```

打開 `http://localhost:5173/`，應該會看到：

- 左側 Sidebar（Dashboard / Purchase Requests / Create Request / Audit Logs）
- Topbar 右上角綠色 `API Healthy` + `BC: mock`
- Dashboard 上有 4 筆 seed data 的統計卡

## 4. 用 Docker Compose 啟動（不想裝 Python / Node 的人）

### 4.1 自己 build（預設）

```powershell
cd C:\Users\Xu\D365\Business-Central-Integration-Suite
docker compose up --build
```

第一次 build 約 3–6 分鐘。會啟動三個服務：

| 服務 | URL | 說明 |
|------|-----|------|
| `db` | `localhost:5432` | PostgreSQL 16 |
| `api` | `http://localhost:8000` | FastAPI |
| `web` | `http://localhost:5173` | nginx 靜態網站 |

要停止：`Ctrl+C`，然後 `docker compose down`。
要清乾淨資料：`docker compose down -v`。

### 4.2 直接拉 GHCR 上的釋出版（更快，不用本地 build）

CI/CD 會自動把每個進到 `main` 的 commit 推到 GitHub Container Registry：

- `ghcr.io/xu323/business-central-integration-suite/api`
- `ghcr.io/xu323/business-central-integration-suite/web`

> 如果 package 還是 private（首次 push 預設），先做一次：
> 1. 在 [github.com/settings/tokens](https://github.com/settings/tokens) 建立一個 classic PAT，勾 `read:packages`。
> 2. PowerShell：`echo <PAT> | docker login ghcr.io -u xu323 --password-stdin`
> 3. 或到 GitHub → Profile → Packages → 該 package → Package settings → Change visibility → Public，之後就不必登入。

跑：

```powershell
cd C:\Users\Xu\D365\Business-Central-Integration-Suite
docker compose -f docker-compose.ghcr.yml up
# 鎖到特定版本：
$env:IMAGE_TAG = "sha-abc1234"; docker compose -f docker-compose.ghcr.yml up
$env:IMAGE_TAG = "1.0.0";       docker compose -f docker-compose.ghcr.yml up
```

## 7. 怎麼操作前端（Demo 動作）

打開 `http://localhost:5173/`：

1. **Dashboard** — 顯示總請購數、各狀態筆數、High-Risk 數量、Open / Synced 金額。
2. **Purchase Requests** — 列出所有請購單，可依狀態篩選、搜尋。
3. **Create Request**（左側 ➕）：
   - 填 `Description`、`Vendor`、加入 1 筆以上 Line。
   - 按「Save as Draft」→ 跳到該筆詳情頁。
4. **Detail** 頁右側「Workflow Actions」：
   - `📨 Submit` —— Draft → Submitted（必須有至少 1 條 Line）
   - `✓ Approve` —— Submitted → Approved（會記下 approver / 時間）
   - `✗ Reject` —— Submitted → Rejected
   - `🔄 Sync to Business Central` —— Approved → Synced；mock 模式會回傳 `PO-MOCK-PR…` 當 BC Document ID
5. **Audit Logs** — 全系統的稽核軌跡，包含 mock sync 成功 / 失敗。

> 金額 ≥ 100,000（在 `.env` 的 `HIGH_RISK_THRESHOLD`）會自動標記為 High-Risk badge。

## 8. 怎麼理解 AL Extension（沒有 BC sandbox 也能讀懂）

打開 [`extensions/bc-procurement/`](extensions/bc-procurement/)，逐檔對應：

| 檔案 | 角色 |
|------|------|
| `app.json` | Extension manifest（id, publisher, idRanges 50100–50199, runtime） |
| `src/enums/PurchaseRequestStatus.al` | 狀態列舉 Draft/Submitted/Approved/Rejected/Synced |
| `src/tables/PurchaseRequestHeader.al` | 主檔（單號、廠商、總金額、High-Risk 旗標） |
| `src/tables/PurchaseRequestLine.al` | 明細表（品項、數量、單價、金額） |
| `src/pages/PurchaseRequestList.al` | BC 內看得到的清單畫面 |
| `src/pages/PurchaseRequestCard.al` | 單筆編輯畫面 |
| `src/pages/PurchaseRequestSubform.al` | 嵌在 Card 裡的明細子畫面 |
| `src/codeunits/ApprovalMgt.al` | **核心商業邏輯**（狀態機 + 高風險判定 + IntegrationEvent） |
| `src/codeunits/BCSyncMgt.al` | 把 Approved 請購單轉成 Purchase Order |
| `src/api/PurchaseRequestAPI.al` | 對外 RESTful API（OData v4） |
| `test/PurchaseRequestTests.al` | 7 個 GIVEN/WHEN/THEN 測試案例 |

> Tip：FastAPI 後端的 `app/routers/purchase_requests.py` 是 AL `ApprovalMgt.al` 的「Python 對映」，兩端使用同樣的狀態機，分別在 BC 內部與對外整合層各實作一份。

## 9. 未來怎麼接「真實的」 Business Central Sandbox

1. **申請 BC sandbox**：到 https://aka.ms/businesscentral 註冊試用，建立一個免費 Sandbox 環境。
2. **註冊 Entra ID App**：到 Azure Portal → App registrations → New registration → API permissions 加 `Dynamics 365 Business Central`（Application permission `app_access`）→ Grant admin consent。
3. **在 BC 裡綁定 App**：BC 內搜尋「Microsoft Entra Applications」(page 9087)，新增一筆，貼 App ID，指派 PermissionSet 為 `D365 BUS PREMIUM` 或自訂。
4. **取 Tenant Id / Company Id / Client Id / Client Secret**，填到 `.env`：
   ```env
   BC_MODE=real
   BC_TENANT_ID=<azure tenant id>
   BC_ENVIRONMENT=Sandbox
   BC_COMPANY_ID=<bc company guid>
   BC_CLIENT_ID=<entra app id>
   BC_CLIENT_SECRET=<entra app secret>
   ```
5. **重啟 backend**：再按 `Sync to Business Central`，請求會走真實 OAuth → BC 的 `purchaseRequests` Custom API。

> 詳細安裝、發行 AL Extension 的步驟在 [`docs/architecture.md`](docs/architecture.md) 第 6 節「真實 BC 串接」。
